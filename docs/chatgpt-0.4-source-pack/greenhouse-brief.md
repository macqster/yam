# YAM Greenhouse Brief

## Core Premise

The greenhouse is a future YAM simulation space for observing, growing,
testing, and curating terminal-native organisms.

It should feel like:

- a contained botanical lab
- a nursery for future scene organisms
- a calm inspectable greenhouse place
- a world-space environment rather than a settings surface

The current first mental image should be:

- botanical-lab discipline
- sparse propagation-room clarity
- careful placement
- calm inspection
- later conservatory expansion only after the base contract proves itself

## Current First-Pass Rule

The first greenhouse pass is functional-space-first.

The ordering rule is:

`room -> access paths -> zones -> fixtures -> planting sites -> environment profile -> inspection surface`

Plant-system-first work is explicitly deferred until the functional-space
contract is stable.

This is the key ordering rule ChatGPT should respect.

## First-Pass Identity

The current bias is:

- one nursery or propagation room
- planning identity: `greenhouse_nursery`
- sparse visual/readability discipline
- symbolic environment
- read-only inspection
- curation-oriented progression rather than gameplay

Useful first-pass shell/support vocabulary:

- `glass_frame`
- `propagation_bench`
- `left_tray`
- `cutting_jar_slot`
- `inspection_marker`
- `training_frame`
- `specimen_shelf`
- `substrate_strip`
- `lamp_zone`

Useful first-pass visual discipline:

- leave empty air around supports
- prefer one or two strong support clusters over dense fill
- treat shell elements as controlled container cues, not decorative wallpaper
- keep labels and markers compact

## Broader Expansion Shape

The preserved longer-range idea is that greenhouse may later widen into a
stable frame containing switchable internal labs or rooms.

Candidate later labs:

- Propagation Lab
- Climate Lab
- Grow Bay Lab
- Vines Lab
- Utility Lab
- Archive / Specimen Lab

Important constraint:

Do not assume this broader frame-plus-labs structure should appear in runtime
before the first inert nursery room is owned cleanly in data.

Good ChatGPT behavior:

- preserve the later lab vocabulary
- avoid treating later labs as immediate runtime requirements
- propose staged widening rather than instant greenhouse sprawl

## Identity Hierarchy

This hierarchy matters and should stay distinct:

- greenhouse identity
- lab or room identity
- planting-site or bay identity
- support identity
- organism identity
- warning identity
- journal-link identity

Support identity and organism identity must remain separate.

This matters because later inspection, warnings, journaling, and transfer logic
need stable targets that do not collapse into one label.

## Environment Bias

The first environment model should stay symbolic and room-level, using compact
fields such as:

- light
- humidity
- temperature
- water
- airflow
- substrate
- outside-weather influence

Outside weather should be disabled by default in the first pass.

## Journaling Bias

Future journaling is valid, but not as the first heavy UI surface.

Journal-target order should be:

1. organism
2. support
3. planting site or bay
4. room or lab
5. greenhouse

The first runtime prep should stay closer to read-only inspection targets than
to a full journal application.

Good early ChatGPT output here:

- marker vocabulary
- short inspection modes
- journal-entry category ideas
- tiny observation text examples

Less useful early output:

- full application-like journaling workflows
- big log browsers
- record-management UI systems

## Bounded Reference Posture

These references are valid, but bounded:

- `cbonsai`: mood, slow emergence, terminal-organism feel
- `HighGrow`: compact room grammar, small bay count, inspect precedent
- `Viridi`: calm care mood and gentle attachment language
- `asciiquarium`: ambient shared-ecosystem atmosphere

These references should not be treated as mandates for:

- crop optimization loops
- harvest gameplay
- detailed scientific plant simulation
- early persistence or economy systems
- dashboard-first greenhouse UI
